# 보안 및 컴플라이언스 가이드 (Security & Compliance)

> Presentation Evaluator v1.0 — 금융기관 내부통제·정보보호 관점 보안 문서

---

## 1. 보안 설계 원칙

본 시스템은 다음 5대 원칙에 기반하여 설계되었습니다:

| 원칙 | 적용 방안 |
|------|----------|
| **최소권한** | 관리자/평가자 역할 분리, 평가자는 본인 평가만 접근 |
| **격리** | 외부 통신 차단, 폐쇄망 단독 동작 |
| **감사가능성** | 모든 주요 액션에 대한 AuditLog 기록 |
| **무결성** | 집계 확정 후 잠금(Lock) 처리, 위·변조 방지 |
| **데이터 최소 수집** | 평가자 정보는 운영 필수 항목만 (이름, 부서) |

|----|------|--------|--------|------|

| T02 | 평가 점수 위·변조 | 高 | 低 | 집계 잠금, 감사 로그 |

## 3. 데이터 보호

### 3.1 저장 데이터 (Data at Rest)

| 데이터 | 저장 위치 | 보호 방식 |
|--------|----------|----------|
| 설명회 정보 | Hive box (`sessions.hive`) | OS 파일시스템 권한 + 디바이스 암호화 |
| 평가 결과 | Hive box (`evaluations.hive`) | 동일 |
| 평가자 정보 | Hive box (`evaluators.hive`) | 동일 |
| 감사 로그 | Hive box (`audit_logs.hive`) | 동일, **삭제 불가 정책** 적용 권장 |
| 관리자 비밀번호 | Hive box (`admins.hive`) | ⚠️ MVP는 평문 저장 → **운영 시 해싱 필수** |

### ⚠️ 운영 전 필수 조치
- [ ] **관리자 비밀번호 bcrypt/argon2 해싱** (현재 MVP 미적용)
- [ ] **Hive 파일 암호화** — `Hive.openBox(..., encryptionCipher: HiveAesCipher(key))` 적용
- [ ] **암호화 키 관리** — OS 키체인(Android Keystore / Web Crypto API) 연동
- [ ] **백업 파일 암호화** — 외부 백업 시 별도 암호화

### 3.2 전송 데이터 (Data in Transit)

**현재 상태**: 외부 통신 없음 (Hive 로컬 저장만)

**향후 백엔드 연동 시 적용 사항**:
- [ ] TLS 1.3 강제 (사내 인증서 사용 가능)
- [ ] 인증서 핀닝 (`http` 패키지 + custom HttpClient)
- [ ] mTLS (mutual TLS) 검토 — 금융권 표준
- [ ] 요청 페이로드 압축 + 무결성 검증 (HMAC)

## 4. 인증 및 인가

### 4.1 관리자 인증 (현재 MVP)

```
Login Flow:
  1. adminId + password 입력
  2. Hive에서 관리자 조회
  3. password 비교 (⚠️ 평문 — 운영 시 해싱 필요)
  4. 성공 시 AppProvider.currentAdmin 설정
  5. AuditLog 기록 (LOGIN action)
```

### 4.2 향후 강화 방안 (Phase 2)

| 단계 | 조치 |
|------|------|
| **Phase 2.1** | bcrypt 해싱 적용 (`crypto` 패키지) |
| **Phase 2.2** | 사내 AD/LDAP 연동 (RADIUS 또는 SAML) |
| **Phase 2.3** | MFA (TOTP 또는 사내 OTP 서버) |
| **Phase 2.4** | SSO (Keycloak 등 사내 IdP) |

### 4.3 평가자 인증

- 평가자별 **6자리 인증 코드** 또는 **QR 코드** 발급
- 1개 세션 = 1개 코드, 제출 후 재사용 불가
- 코드 입력 시점에 세션 상태 검증 (ongoing 인지)

### 4.4 역할 기반 접근 제어 (RBAC)

| 역할 | 권한 |
|------|------|
| **Super Admin** | 모든 기능 + 다른 관리자 관리 |
| **Admin** | 설명회/템플릿/평가자 CRUD, 결과 잠금 |
| **Viewer** | 결과 조회만 가능 (감사 대비) |
| **Evaluator** | 본인 할당 세션의 평가 입력만 |

## 5. 감사 추적 (Audit Trail)

### 5.1 기록 대상 액션

| 카테고리 | 액션 | 기록 항목 |
|---------|------|----------|
| 인증 | LOGIN, LOGOUT, LOGIN_FAILED | actorId, timestamp, ip(N/A) |
| 설명회 | CREATE, UPDATE, DELETE, LOCK | sessionId, before/after |
| 템플릿 | CREATE, UPDATE, DELETE | templateId, before/after |
| 평가자 | CREATE, UPDATE, DEACTIVATE | evaluatorId |
| 평가 | SUBMIT, EDIT (관리자 강제) | evaluationId |
| 내보내기 | EXPORT_CSV, EXPORT_PDF | sessionId, exportedBy |

### 5.2 감사 로그 무결성

**현재 구현**:
- Hive box에 append-only 방식으로 기록
- UI에서 삭제 기능 미제공

**향후 강화**:
- [ ] 해시 체인 (이전 로그 해시 포함) — 위·변조 즉시 탐지
- [ ] 디지털 서명 (관리자 개인키)
- [ ] WORM 스토리지 연동 (사내 Object Storage)

### 5.3 로그 보존 기간

| 데이터 종류 | 권장 보존 기간 | 근거 |
|------------|---------------|------|
| 감사 로그 | **5년** | 금융업감독규정 시행세칙 |
| 평가 결과 | **3년 이상** | 내부통제 자료 |
| 평가자 개인정보 | **1년 (행사 종료 후)** | 개인정보보호법 |

## 6. 개인정보보호

### 6.1 수집 항목

| 항목 | 수집 목적 | 보유 기간 |
|------|----------|----------|
| 평가자 이름 | 평가자 식별 | 행사 종료 후 1년 |
| 부서/소속 | 평가자 검증 | 동일 |
| 평가 코멘트 | 평가 결과 활용 | 3년 |

⚠️ **수집하지 않는 항목**: 주민번호, 연락처, 이메일, 위치정보

### 6.2 개인정보보호법 준수

- [ ] **수집·이용 동의** — 평가자 첫 진입 시 동의 화면 추가 필요
- [ ] **목적 외 이용 금지** — 평가 목적 외 사용 금지 명시
- [ ] **파기 절차** — 보유 기간 경과 시 자동 삭제 스크립트
- [ ] **개인정보 처리방침** — 사내 공시 페이지 게시
